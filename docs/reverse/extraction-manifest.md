# Extraction Manifest

Generated by `tools/hexlab` into `artifacts/disasm`.

## Artifacts
- `triage.json` - machine-readable triage metadata and split heuristics
- `signature_offsets.txt` - known marker offsets
- `strings_min6.txt` - extracted printable strings (min length 6)
- `segments/code_candidate.bin` - likely code-heavy region
- `segments/assets_candidate.bin` - likely asset/data-heavy region
- `segments/code_candidate.hex.txt` - text hexdump of candidate code segment
- `segments/assets_candidate.hex.txt` - text hexdump of candidate asset segment
- `segments/code_candidate.strings.txt` - printable strings from code segment
- `segments/assets_candidate.strings.txt` - printable strings from assets segment
- `transpile/sopwith_image.asm` - full binary as assembly `db` image for transpile tooling
- `decoded/index.json` - second-pass typed candidate index (palettes/sprites/tables)
- `decoded/palettes/*` - palette candidate JSON + raw + swatch PPM
- `decoded/sprites/*` - sprite candidate JSON + raw + sprite-sheet PGM
- `decoded/tables/*` - table candidate JSON + raw + TSV view
- `decoded/remaining_items.json` - third-pass inventory of untyped runs before WASM conversion

## Transpile readiness
- A full source image is available in assembly form (`sopwith_image.asm`).
- Candidate code and data blobs are separated for focused translation passes.
- Known string anchors can seed symbol naming in decompiler projects.
- Candidate asset blocks are typed heuristically into palette/sprite/table buckets.
- Remaining unknown runs are tracked explicitly for follow-up decoding.

## Known gaps
- No verified instruction-level disassembly map yet.
- Asset typing is currently heuristic and must be validated against runtime use.
- Music/event stream formats remain unresolved in remaining-items inventory.
- Entry-point certainty depends on static tool setup and runtime traces.
